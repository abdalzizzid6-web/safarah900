## 4. Duplicate & Redundant Logic

### 4.1 Duplicate Services
- **`src/services/matchService.ts` vs `src/core/repository/MatchesRepositoryV2.ts`**: `matchService` acts as a middleman that often simply forwards calls to `MatchesRepositoryV2`.
- **`src/hooks/useFootballApi.ts` vs `src/hooks/useMatchesV2.ts`**: Both provide `useLiveMatches`, `useFixtures`, `useLeagues`. `useFootballApi` is legacy (non-React Query) while `useMatchesV2` is modern.

### 4.2 Redundant Normalization
- **`src/core/utils/matchNormalization.ts`**: Client-side normalization.
- **`server/utils/normalizer.ts`**: Server-side normalization.
- *Issue:* Logic is duplicated between client and server, leading to potential drift in how a "Match" object is shaped.

## 5. Cache Layer Analysis

| Layer | Implementation | Scope |
|-------|----------------|-------|
| **Server Cache** | In-memory (`proxyCache`, `matchSsoCache`) in `server/index.ts` | API Proxy results |
| **Firestore Cache** | `server/firestore/cache.ts` -> `matches.json` | Aggregated match data for high-traffic pages |
| **Client Memory** | `src/core/api/cacheManager.ts` | In-memory Repository caching |
| **Client Durable** | `localStorage` (via `cacheManager`) | Persistence across sessions |
| **React Query** | `src/hooks/useMatchesV2.ts` | Component-level state caching |

## 8. Impact Analysis (What if deleted?)

- **`matchService.ts`**: Total breakdown of `useFootballApi` and `useMatchesV2`. Nearly all match-related UI will fail to load.
- **`matchMapper.ts`**: Only `teamApi.js` (legacy) will fail. Minimal impact on modern components.
- **`useFootballApi.ts`**: `LeaguePage.tsx` and `LiveMatches.tsx` will fail. These components have not yet migrated to `useMatchesV2`.
